# 宝可梦“我是谁”图片生成器设计规格
> 请注意本文档由LLM AI生成。
## 概述
本项目提供一个可部署到 GitHub Pages 的纯前端图片生成器。用户可以指定宝可梦，或由系统随机选择一只宝可梦，再以剪影或区域裁剪方式生成“我是谁”题面图及对应答案图。
第一版面向公开、免费、无广告的非商业使用场景。它不包含答题、计分、用户账户、服务端存储或商业授权能力。宝可梦名称、形象、标志和用户提供的题面模板可能受到权利方保护；页面必须标注非官方、非商业、素材来源及权利归属，但这些声明不构成授权。
## 设计思路
核心矛盾是：网页需要覆盖大量宝可梦形态并稳定导出图片，但纯静态部署不能依赖服务端抓取，浏览器实时枚举 PokéAPI 又会引入接口波动、请求量和跨域问题。
因此，数据更新与网页使用被分离。维护者通过脚本显式同步 PokéAPI 数据，审计后将版本化清单提交到仓库；网页只读取随版本发布的静态清单。图片使用固定到特定提交版本的 PokéAPI sprites CDN 地址，并通过候选图片回退提高形态覆盖率。
生成逻辑与界面也保持分离。选择、随机和裁剪参数由不依赖 React 的核心模块管理，Canvas 渲染器只负责把确定的生成状态绘制为题面或答案。这使随机规则、像素处理和裁剪边界可以脱离页面进行测试。
## 产品范围
### 宝可梦选择
指定选择支持简体中文名、英文名和全国图鉴编号搜索。用户先选择物种，再选择该物种下的具体形态。
随机选择采用两阶段等概率规则：先从可用物种中等概率选择一个物种，再从该物种的可用形态中等概率选择一个形态。这样不会因为某个物种拥有较多形态而显著提高其出现概率。
第一版的“全部形态”定义为：PokéAPI 中存在形态记录，并且能够解析出至少一张可加载正面图片的物种变种或外观形态。缺少可用图片的形态不进入随机池，并在数据审计报告中列出；系统不得使用基础形态图片冒充缺图形态。
### 生成模式
剪影模式读取原图透明通道，将所有非透明像素绘制为纯黑色，同时保留透明边缘。剪影在题面模板左侧框内按比例完整居中。
区域裁剪模式先计算原图非透明像素边界，再以完整居中缩放比例的 1.5～3.0 倍随机产生位移与缩放参数。有效结果需要让非透明像素覆盖至少 15% 的内容区域，同时最多展示源图 70% 的非透明像素。用户可以拖动图片、使用滑杆缩放，也可以要求系统再次随机裁剪。框外内容通过 Canvas 裁剪区域隐藏。
### 题面与答案
题面和答案都以用户提供的 1024×768、4:3 图片为模板，输出尺寸保持 1024×768，不进行额外放大。
题面左框显示剪影或局部彩色图片，右侧保留模板自带的“???? 我是谁”文字。答案左框显示完整彩色图片；右侧使用与背景匹配的蓝色区域覆盖问题文字，再以黄色文字绘制简体中文名、英文名、全国图鉴编号和形态名。
两张图片均通过 `canvas.toBlob("image/jpeg", 0.92)` 导出。文件名包含全国图鉴编号、英文标识及 `question` 或 `answer`，例如 `0025-pikachu-question.jpg`。
## 界面设计
桌面端采用工作台式双栏布局。左栏集中放置搜索、物种与形态选择、随机选择、生成模式、裁剪参数和重新生成操作；右栏持续显示 4:3 预览、题面/答案切换及两个下载按钮。
窄屏设备改为上下排列，控制区位于预览区之前。预览仅按 CSS 缩放，实际 Canvas 始终以 1024×768 绘制，避免页面尺寸影响下载结果。
未选中宝可梦时禁用模式参数和下载操作。图片仍在加载、数据加载失败或导出失败时，界面必须显示当前状态，不能下载空白或不完整图片。

## 数据设计
静态清单按物种组织，避免随机选择时重复扫描所有形态。每个物种记录全国图鉴编号、英文标识、多语言名称和形态列表；每个形态记录形态标识、显示名称、默认/特殊形态标记及按优先级排列的图片候选。
```ts
interface PokemonSpeciesRecord {
  id: number;
  slug: string;
  names: { zhHans: string; en: string };
  forms: PokemonFormRecord[];
}

interface PokemonFormRecord {
  id: string;
  slug: string;
  names: { zhHans?: string; en: string };
  isDefault: boolean;
  gender: "unspecified" | "male" | "female";
  flags: {
    isMega: boolean;
    isBattleOnly: boolean;
    isCosmetic: boolean;
  };
  imageCandidates: string[];
}
```
同步脚本遍历 `pokemon-species`、物种下的 varieties、对应 `pokemon` 资源及其 `pokemon-form` 资源，以同时覆盖影响属性的变种和仅影响外观的形态。当 sprite 数据提供独立的雌性图片时，脚本额外生成 `gender: "female"` 的形态记录；默认图片使用 `gender: "unspecified"`。图片候选依次采用官方立绘、Pokémon HOME 图片和普通正面图。
形态缺少简体中文名称时，界面回退到英文形态名；物种简体中文名称缺失属于数据审计错误。同步结果包含 PokéAPI 数据版本、sprites 提交版本、生成时间和来源信息。
52Poké 不作为第一版运行时或构建时数据源。其内容采用 CC BY-NC-SA 3.0，机器读取还要求可识别的 User-Agent、串行请求和超过 500 毫秒的请求间隔；当前需求可由 PokéAPI 满足，无需引入额外许可与抓取约束。
## 数据与生成流程
维护者执行同步命令后，脚本拉取并规范化数据、解析图片候选、生成静态清单，再执行唯一性、关系和完整性审计。审计失败时不得覆盖上一版有效清单。
用户打开网页后，应用只加载静态清单。指定选择或物种优先随机都会产生一个确定的物种与形态；图片加载器依次尝试候选地址。加载成功后，选择核心创建生成状态，Canvas 渲染器据此刷新题面和答案预览。
区域裁剪模式的初始参数由可注入的随机数生成器产生，用户调整后保存为当前状态。题面与答案共享同一宝可梦记录，但答案始终使用完整彩色图，不复用题面的剪影或裁剪参数。
## 异常处理
静态清单无法加载时显示阻断式错误和重试入口，不显示空随机池。搜索无结果时保留当前选择并提示用户修改关键词。
图片候选加载失败时继续尝试下一候选；全部失败时标记该形态在当前会话不可用，阻止预览和下载，并允许重新选择。Canvas 因跨域污染、编码失败或浏览器不支持而无法导出时，界面展示具体失败原因。
随机裁剪最多尝试 20 次寻找满足有效像素约束的结果；仍未找到时使用完整居中比例的 2 倍缩放，并将非透明边界中心对齐内容区域中心。任何回退都不得让内容区域完全空白。
## 验证方式
Vitest 单元测试覆盖物种优先随机、物种内形态随机、名称回退、图片候选回退、剪影像素转换、裁剪边界、模板布局和文件命名。随机相关测试使用可注入的固定随机序列，避免概率性失败。
数据审计验证物种编号与形态标识唯一、形态归属有效、每个可用物种至少存在一个可用形态、物种中文名存在、图片地址格式有效，并输出缺图或名称回退报告。
Playwright 在真实浏览器中验证指定选择、随机选择、剪影、裁剪拖动与缩放、题面/答案切换、JPG 类型与尺寸，以及桌面双栏和移动端上下布局。Canvas 测试至少检查关键像素区域，防止导出空白图片。
GitHub Actions 在部署前依次执行类型检查、单元测试、数据审计、浏览器测试和生产构建。部署流程只消费仓库中的版本化数据，不在 CI 中重新访问 PokéAPI。
## 素材与部署约束
用户提供的背景图在实现时保存为可替换模板资源。模板包含宝可梦官方标志及相关美术元素，仓库和页面需要保留素材来源、非官方和非商业声明；如果公开托管方或权利方提出要求，应能通过替换单个模板文件及布局配置移除该素材。
静态站点使用相对基础路径，确保 GitHub Pages 子路径部署后，清单、模板、字体和应用资源仍能正常加载。远程图片 CDN 必须支持匿名 CORS，否则 Canvas 无法安全导出。
